# 04. Как оценивать execution paths — рамка, а не ответ

> **Цель файла:** дать тебе **систему вопросов**, через которые ты сам сравнишь любой execution path с любым другим. Не «вот моя decision-table», а «вот критерии, по которым ты построишь СВОЮ».

---

## 4.1 Шесть классов исполнения

Любую compute-задачу можно отнести к одному (или нескольким) классам:

```text
1. Synchronous SQL/NoSQL  — один запрос к одному engine, ждём результат
2. Multi-stage pipeline   — DAG из независимых шагов с зависимостями
3. Index-based serving    — заранее построенная структура для быстрых ops
4. Cache lookup           — предыдущий результат, годный для использования
5. Precomputed materialize— заранее посчитанные часто запрашиваемые куски
6. Incremental / streaming— только дельта между состояниями
```

Это не «выбери один». Реальные системы используют комбинации. Скажем, для одной аудитории — cache hit; для другой — multi-stage pipeline; третья — index intersect плюс sync SQL post-filter.

## 4.2 Критерии для оценки любого пути

Это **рамка**. Применяй её к КАЖДОМУ пути, который рассматриваешь:

### A. Latency profile
```text
- Best case  (cold cache, наихудшая нагрузка) ≈ ?
- Typical    (средние данные, средняя нагрузка) ≈ ?
- Worst case (deep DAG, max payload, peak load) ≈ ?
```

### B. Cost profile
```text
- CPU      (compute hours)
- Memory   (peak working set)
- IO       (disk reads, network bytes)
- Storage  (что нужно хранить и как долго)
```

### C. Correctness guarantees
```text
- Точный результат vs приближённый
- Reproducibility (через 3 дня тот же вывод?)
- Какие edge-cases ломают (NULL, range, multi-source)?
- Drift (каков потолок staleness)?
```

### D. Operational complexity
```text
- Кто хозяин этого пути в инфре?
- Сколько runbooks нужно?
- Какие новые failure modes появляются?
- Observability — что мы увидим, что не увидим?
```

### E. Boundary conditions
```text
- Когда путь хорошо работает?
- Когда плохо?
- Где degradation мягкая (slowly worse), где жёсткая (cliff)?
```

### F. Roll-out / change cost
```text
- Какие контракты нужно поменять?
- Какие downstream-команды нужно вовлечь?
- Какие миграции данных?
- Что нельзя откатить?
```

### G. Coupling
```text
- Что становится связаннее (новая зависимость)?
- Что остаётся независимым?
```

## 4.3 Правильные вопросы при сравнении

Не «какой путь лучше», а:

```text
ДЛЯ ДАННОГО КЛАССА АУДИТОРИЙ
И ДАННОГО ПРОФИЛЯ НАГРУЗКИ
И ДАННЫХ БИЗНЕС-ОГРАНИЧЕНИЙ —

какой путь в каких пределах характеристик A-G?
```

«Класс аудиторий» определяешь ты — например, по форме predicates, размеру результата, частоте refresh, downstream-у. Без определения класса decision-table бессмысленна.

---

## 4.4 Анти-паттерны при сравнении путей

```text
1. Сравнивать "среднее" без дисперсии.
   p50 = X, p95 = 10x  — это два разных мира.

2. Сравнивать на разных нагрузках.
   Bench пути A на пустой системе и пути B на peak — нечестно.

3. Игнорировать correctness ради скорости.
   "В 5 раз быстрее" + "иногда другой результат" = бесполезно.

4. Игнорировать operational tail.
   Путь, который требует 5 новых runbooks, — не "быстрее" в долгую.

5. Сравнивать новый путь с идеализированной версией старого.
   Старый путь имеет accumulated tweaks за годы. Новый — ещё нет.

6. Не учитывать рост нагрузки.
   Что станет с путём, если объём данных вырастет 10x?
```

## 4.5 Чем заполняешь decision-table сам

Возьми любую compute-задачу из твоего проекта. Заполни СВОЮ таблицу:

```md
| Критерий              | Path A | Path B | Path C |
|-----------------------|--------|--------|--------|
| Best/typical/worst    |        |        |        |
| CPU / Mem / IO / Stor |        |        |        |
| Correctness invariant |        |        |        |
| Reproducibility       |        |        |        |
| Boundary (cliff)      |        |        |        |
| Roll-out cost         |        |        |        |
| Coupling delta        |        |        |        |
```

Если хотя бы одна клетка пустая — у тебя ещё **нет** оснований выбирать. Идёшь экспериментировать.

---

## 4.6 Что изучить, чтобы заполнять эту таблицу честно

| Тема | Что даёт | Источник |
|---|---|---|
| Queueing theory + Little's law | latency = arrivals × service time / capacity | Harchol-Balter |
| Statistics для бенчей | p50/p95/p99, дисперсия, доверительные интервалы | блог brendangregg.com |
| Database query plans | EXPLAIN / EXPLAIN ANALYZE / PIPELINE | SQL Performance Explained |
| OLAP vs OLTP storage | row vs columnar, partition, compression | DDIA гл. 3 |
| Data structures для intersection | bitmap (Roaring), inverted index | Roaring docs, Lucene paper |
| Caching theory | hit rate, eviction, invalidation, stampede | Designing Data-Intensive Applications + блоги Cloudflare |
| Streaming windows + watermarks | event time, processing time, watermark | Streaming Systems book |

## 4.7 Упражнения

```text
УПРАЖНЕНИЕ A. Возьми ОДНУ конкретную аудиторию из staging (с реальными данными).
              Заполни таблицу 4.5 для:
                - текущий путь (DAG в Target);
                - синхронный CH SQL;
                - cache (предположим hit).
              Где нет данных — пометь "НЕ ИЗМЕРЕНО ЕЩЁ".

УПРАЖНЕНИЕ B. Найди такой класс аудиторий, где бы у тебя ВСЕ критерии в таблице
              сильно отличались между путями. Это и есть кейс, где выбор пути имеет смысл.

УПРАЖНЕНИЕ C. Найди класс аудиторий, где разница между путями небольшая — там выбор
              пути не важен; тратить время не нужно.

УПРАЖНЕНИЕ D. Сформируй СВОЮ гипотезу: какой путь подходит каким аудиториям, почему,
              где она опровергается экспериментом.
```

---

## 4.8 Где это может стать ошибкой

```text
- Ты заполнишь таблицу впечатлениями вместо чисел → решение основано на интуиции, а не данных.
- Ты будешь считать только compute, забыв operational/coupling → выкатишь и больно поймаешь.
- Ты не будешь возвращаться к таблице, когда нагрузка вырастет → решение устареет.
- Ты будешь думать, что выбор — это binary; реально — gradient (X% трафика по пути A).
```
