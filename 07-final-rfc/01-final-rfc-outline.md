# 01. Как ты сам напишешь свой RFC

> **Цель файла:** объяснить, как структурно подойти к RFC, и **дать пустой каркас**. Здесь сознательно нет ни моих фаз, ни моих чисел, ни моих рекомендаций. Это твоя инициатива — ты её и наполняешь.

---

## Когда RFC уместен

Не пиши RFC, если:
- Ты ещё не знаешь, на какой класс задач предлагается изменение.
- У тебя нет численного baseline-а.
- Ты не понимаешь существующих гарантий, которые можешь сломать.
- Ты не разговаривал с владельцем хотя бы одной затронутой системы.

Признаки готовности к RFC:
- Ты можешь за 10 минут на доске объяснить **текущую** архитектуру end-to-end.
- У тебя есть числа (compute / export / delivery / freshness и т.п.).
- У тебя есть **гипотеза** (твоя), которая сужает круг изменений до конкретного класса задач.
- У тебя есть «proof of life» эксперимент, который её **подтверждает или опровергает**.
- Ты прошёл 70%+ self-check вопросов из `06-practice/04-self-check-questions.md`.

## Как RFC рождается из learning-os

```text
01-context/  → факты из кода (стартовый ландшафт)
02-architecture/ → frameworks мышления (как ты будешь рассуждать)
04-learning-tracks/ → теория, которая нужна, чтобы рассуждать корректно
05-weekly-sprints/ → 8 недель погружения через действие
06-practice/ → эксперименты, на которых ты соберёшь свои числа

→ Из этих слоёв ты ВЫВОДИШЬ свою гипотезу.
→ Гипотеза + numbers + альтернативы = RFC.
```

Если ты пропустил какой-то слой — RFC выйдет хрупким. Не торопись.

## Каркас RFC (пустой)

Используй или свой шаблон из `06-practice/03-rfc-template.md`. Ниже — суть, не скрипт:

### 1. TL;DR
Одно-два предложения. Что и зачем. Если читатель прочёл только это — он понимает, в чём суть.

### 2. Problem
Что мы наблюдаем (с числами). Кому больно. Почему сейчас.
*Предупреждение:* «нам кажется, что медленно» — не проблема. «p95 для X-класса = Y, при том что бизнес требует Z» — проблема.

### 3. Current state
Кратко — как сейчас. Со ссылками file:line. Без суждений.

### 4. Diagnosis
**Твои** наблюдения, основанные на **твоих** замерах. Где время, IO, ошибки. Какие связи между симптомами.

### 5. Альтернативы
Каждая — с trade-off. Используй рамку из `02-architecture/04-graph-vs-sql-vs-bitmap-vs-cache.md` (А-G критерии). НЕ перечисляй мои 6 путей по умолчанию — перечисляй **те**, которые имеют смысл для **твоей** проблемы.

### 6. Recommendation
Что предлагаешь. Почему именно так. **Что не предлагаешь** (и почему).

### 7. Roll-out
Какие фазы, какие feature flags, какой shadow-режим, какой cohort. Без воображаемых сроков — с честными «requires X person-weeks».

### 8. Rollback
Какие триггеры. Как быстро. Что нельзя откатить — ОБЯЗАТЕЛЬНО.

### 9. Success criteria
Какие метрики покажут, что работает. Какие — что не работает.

### 10. Risks
Технические, операционные, организационные. Mitigation для каждого.

### 11. Open questions
Что осталось неясным. Что нужно у кого спросить, чтобы стало ясно.

### 12. Effort
В FTE-weeks. Зависимости. Critical path.

## Ошибки, которые легко допустить (из чужих RFC)

```text
1. Скрытая prescriptiveness:
   "We will introduce X" — а X не обоснован альтернативами.

2. Цели без чисел:
   "Improve performance" — насколько? для какого класса? до какой цифры?

3. Отсутствие downside section:
   Каждое решение что-то ухудшает. Если не написал — readers не доверяют.

4. Roll-out plan = wishful thinking:
   "Phase 1 — 2 weeks, Phase 2 — 4 weeks". Без обоснования времени.

5. Игнорирование владельцев:
   RFC меняет систему X, но автор не говорил с владельцем X.

6. "Уберём X — и всё станет лучше":
   Любая система имеет accumulated tweaks за годы.
   Замена несёт скрытую стоимость.

7. Один RFC решает всё:
   Если PR-аналог в RFC меняет 5 разных вещей — это 5 RFC, не один.
```

## Как готовиться к review

```text
1. Pre-read с тимлидом каждой затронутой системы (1:1, не group call).
   Получи их feedback ПЕРВЫМ. Внеси изменения.

2. Pre-read с PM:
   подтверди, что problem statement соответствует бизнес-боли.

3. Pre-read с SRE:
   roll-out / rollback / observability — это их территория.

4. Только потом — broad review.
```

Если ты сразу несёшь RFC в большой канал — это сигнал, что не было pre-work. Senior-engineer не делает так.

---

## Что произойдёт после approve

```text
Approve ≠ done.

После approve:
- Phase 0: shadow-mode эксперимент 2-4 недели. Учишься на нём.
- Phase 1: production roll-out на cohort. Получаешь numbers.
- Phase 2-N: постепенное расширение, в каждой phase — re-evaluation.
- Не обязательно дойти до Phase N. Можно остановиться, если данные говорят.
```

RFC — не гарантия, что план реализуется в полном объёме. Это разрешение начать.

## DoD на твой RFC

```text
[ ] TL;DR умещается в 2 предложения.
[ ] Все альтернативы с trade-off-ами по 7 критериям из 02-arch/04.
[ ] Все числа реальные (не «по слухам»).
[ ] Numbers собирались по правилам 06-practice/02-benchmark-template.md.
[ ] Pre-read с минимум 3 stakeholders проведён до publish.
[ ] Есть phase 0 (shadow-mode), не сразу production.
[ ] Rollback явный и быстрый.
[ ] Open questions честно перечислены, не «всё ясно».
```

## Когда НЕ публиковать RFC

```text
- Гипотеза опровергнута: запиши почему, не публикуй RFC. Публикуй post-mortem гипотезы.
- Изменилась бизнес-задача: переоценивай весь RFC, не подкручивай частично.
- Stakeholder заблокировал на pre-read: разберись, прежде чем эскалировать.
- Ты не уверен: вернись к спринтам, добери numbers.
```

---

## Альтернатива RFC: ADR

Architecture Decision Record (ADR) — короткий формат, 1-2 страницы. Полезен для:
- Решений, которые ты принимаешь и фиксируешь, без «кросс-командной аппробации».
- Trade-off-ов, выбранных в одной команде.

ADR ≠ RFC. RFC требует approve. ADR — фиксация принятого. Иногда серия ADR заменяет один большой RFC. Подумай, что уместно.

---

## Финал

Твой RFC будет хорошим, если:
1. Я его прочту и не пойму **где ты подсмотрел у меня** — потому что ты не подсматривал.
2. PM прочтёт и скажет: «да, это про нашу боль».
3. Тимлид Target прочтёт и скажет: «это разумно, я бы пошёл этим путём».
4. SRE прочтёт и скажет: «rollback понятен, observability есть».

Если хотя бы одного ответа нет — возвращайся в спринты.
