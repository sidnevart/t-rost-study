## День 1 — Выбор use-case
**Цель:** найти простую аудиторию-кандидата для one-SQL пути.

- Критерии: простой predicate-set, всё в CH, < 1M результат.
- Например: «клиенты Москвы с подпиской PRO».

**Практика:** найти такую аудиторию в проде/staging или построить синтетический params.

**Артефакт:** описание use-case + почему его выбрали.

**DoD:** есть ровно один кейс для всех экспериментов недели.

---

## День 2 — Текущий путь (DAG)
**Цель:** замерить текущий DAG-путь для этого use-case.

- Прогнать стандартный пайплайн.
- Снять числа compute / export / delivery (как в sprint 3).

**Практика:** замер 5 раз (для p50/p95).

**Артефакт:** baseline-таблица.

**DoD:** числа есть.

---

## День 3 — One-SQL путь
**Цель:** написать один CH-запрос, эквивалентный текущему DAG-у.

- Использовать те же CH-таблицы, что и Target.
- Замерить EXPLAIN ANALYZE.
- Замерить wall-clock.

**Практика:** на staging-CH прогнать запрос несколько раз (учесть cache).

**Артефакт:** SQL + план + времена.

**DoD:** результат one-SQL совпадает по count() с DAG-результатом ±0.1%.

---

## День 4 — Cache путь
**Цель:** прикинуть cache-hit потенциал.

- Посчитать на staging БД: сколько уникальных `params_hash` на 10k audience-id-ов.
- Если высокий cache-rate (>30%) — это сильный аргумент.

**Практика:** SQL над `auditory.params` + sha256.

**Артефакт:** табличка cache-hit-rate по типам аудиторий.

**DoD:** есть число «потолок cache hit-rate».
