# LikeC4 Diagrams

[LikeC4](https://likec4.dev) is a code-first DSL for the C4 model. Sources live as `.c4` text files; the CLI generates an interactive site, SVG/PNG exports, and Mermaid/Dot output. Compared to plain Mermaid, LikeC4 gives you:

- **Typed model** — declare elements once, reference them everywhere; renames are safe
- **Multiple views over one model** — Context / Container / Component / Dynamic from the same source
- **Live preview** — `likec4 start` opens an interactive explorer
- **Exports** — SVG / PNG / Mermaid / D2 from the CLI

## When to Use

- User asks for a C4 diagram (System Context, Container, Component, Dynamic, Deployment)
- User says "нарисуй", "обнови диаграмму", "draw the architecture"
- After `flow-extractor` produces flow docs — convert each Dynamic flow into a LikeC4 dynamic view
- Updating an existing LikeC4 model — read `diagrams/*.c4` first, then add/modify
- For a one-off scratch diagram, prefer **inline Mermaid** in the markdown; for a living model that several docs reference, use LikeC4

## Repo Layout

```
diagrams/
├── model.c4              # Elements: actors, systems, containers, components
├── views.c4              # Views: which elements appear in which diagram
├── styles.c4             # Optional shared styles
└── generated/            # Build output — git-ignored, regenerated by CLI
    ├── *.svg
    └── *.png
```

If the project already has a different convention (e.g. `docs/architecture/`), follow it. Don't rearrange existing diagrams.

## Workflow

### Step 1 — Install / verify CLI

```bash
# Project-local (preferred, pinned to package.json):
npm install --save-dev likec4

# Or one-shot via npx (no install needed):
npx likec4@latest --version
```

The skill assumes `npx likec4@latest <command>` works from the repo root. If the project has no `package.json`, that's fine — `npx` will fetch on demand.

### Step 2 — Define / update the model

Edit `diagrams/model.c4`:

```c4
specification {
  element actor
  element system
  element container
  element component
}

model {
  user = actor 'Marketer' {
    description 'Creates audiences and target campaigns'
  }

  audiencePlatform = system 'Audience Platform' {
    description 'Compute, store, and deliver audiences'

    api = container 'API' 'Kotlin / Spring'
    compute = container 'Compute Worker' 'Kotlin' {
      description 'Runs audience definitions against ClickHouse'
    }
    store = container 'Result Registry' 'PostgreSQL'
    bus = container 'Job Queue' 'PostgreSQL LISTEN/NOTIFY'
  }

  target = system 'Target' {
    description 'Delivers audiences to ad platforms'
  }

  user -> audiencePlatform.api 'Creates audience'
  audiencePlatform.api -> audiencePlatform.bus 'Enqueues compute job'
  audiencePlatform.bus -> audiencePlatform.compute 'Delivers job'
  audiencePlatform.compute -> audiencePlatform.store 'Persists result'
  audiencePlatform.store -> target 'Exports audience'
}
```

### Step 3 — Define views

Edit `diagrams/views.c4`:

```c4
views {
  view index of audiencePlatform {
    title 'Audience Platform — System Context'
    include *
  }

  view containers of audiencePlatform {
    title 'Audience Platform — Containers'
    include audiencePlatform.*, target, user
  }

  dynamic view createAudienceFlow {
    title 'Flow: Create Audience'
    user -> audiencePlatform.api 'POST /audiences'
    audiencePlatform.api -> audiencePlatform.bus 'enqueue'
    audiencePlatform.bus -> audiencePlatform.compute 'consume'
    audiencePlatform.compute -> audiencePlatform.store 'write rows'
    audiencePlatform.store -> target 'export'
  }
}
```

### Step 4 — Preview live

```bash
npx likec4@latest start
```

Opens `http://localhost:5173` with an interactive view of every diagram. Useful while iterating.

### Step 5 — Export static assets

For embedding in markdown / PRs / RFCs:

```bash
# All views as SVG into diagrams/generated/
npx likec4@latest export svg -o diagrams/generated

# Single view as PNG
npx likec4@latest export png --view containers -o diagrams/generated

# Mermaid (e.g. for a GitHub README that doesn't render SVG well)
npx likec4@latest export mmd --view createAudienceFlow
```

Add `diagrams/generated/` to `.gitignore` — regenerate, don't commit. Exception: if SVGs need to be embedded in docs viewed without a build step, commit them.

### Step 6 — Embed in docs

In any markdown file:

```markdown
![Audience Platform — Containers](../diagrams/generated/containers.svg)
```

For dynamic views inside a flow doc produced by `flow-extractor`, prefer the LikeC4 SVG over Mermaid once the flow is stable — it stays in sync with the model.

## Best Practices

1. **One model, many views.** Don't duplicate elements across files. If a container appears in three diagrams, it's still one declaration in `model.c4`.
2. **Names match the code.** Element names should match service/module names in the codebase. Searchability matters more than prettiness.
3. **Descriptions are short.** One line per element. Long explanations belong in the doc that embeds the diagram.
4. **Dynamic views per flow.** Each flow from `flow-extractor` → one `dynamic view` block. Keeps flows visualized consistently.
5. **Regenerate on commit.** A pre-commit hook or CI step can run `likec4 export svg` so the SVGs always match the source.
6. **Don't over-style.** Default LikeC4 styles are good. Custom colors should mean something (e.g. external systems = grey).

## Anti-Patterns to Avoid

- Drawing the same architecture in Mermaid AND LikeC4 — pick one source of truth per concept
- Putting business prose in `.c4` files — keep them structural; explanations go in the surrounding doc
- Generating diagrams in CI but never opening `likec4 start` locally — the interactive view catches mistakes the export silently passes
- Committing `diagrams/generated/` AND regenerating on every push — produces noisy diffs

## Quick Reference — DSL Cheatsheet

```c4
specification {
  element <kind>           // declare element kinds (system, container, component, ...)
  tag <name>               // tags for filtering
}

model {
  alias = element 'Title' 'Tech?' {
    description '...'
    tag #external
    -> otherAlias 'relation label'
  }
}

views {
  view <id> of <element> {       // scoped view
    title '...'
    include *
    exclude element.tag #external
  }

  dynamic view <id> {            // sequence-style flow
    a -> b 'step 1'
    b -> c 'step 2'
  }
}
```

Full DSL: https://likec4.dev/dsl/

## Examples

### Example 1: First diagram in a fresh repo
**User**: "Нарисуй C4-диаграмму Audience Platform"
**Action**: Create `diagrams/model.c4` with actors + systems + containers, `diagrams/views.c4` with one Context view + one Container view, run `npx likec4@latest export svg -o diagrams/generated`, embed in `02-architecture/01-audience-platform-model.md`.

### Example 2: Add a dynamic flow to existing model
**User**: "Добавь диаграмму флоу export"
**Action**: Read existing `diagrams/*.c4`, add a `dynamic view exportFlow` block, regenerate SVG, embed in the relevant flow doc.

### Example 3: Convert a flow-extractor output into LikeC4
**Action**: For each `flow-NN-*.md` produced by `flow-extractor`, add a matching `dynamic view` block. The Mermaid sequence diagrams in those docs become LikeC4 dynamic views — single source of truth.
