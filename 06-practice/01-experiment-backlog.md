# 01. Как ты сам проектируешь эксперимент

> **Цель файла:** дать тебе **рамку**, как формировать и проводить experiment, а не список моих экспериментов. Здесь нет «10 вопросов с моими гипотезами» — есть метод, по которому ты будешь придумывать и записывать СВОИ.

---

## A. Почему это важно

Эксперимент — это **способ научиться** в реальной системе. Без него все архитектурные обсуждения остаются мнениями. С ним — у тебя появляется числа и доводы.

**Senior-engineer ≠ тот, кто знает ответ.** Senior-engineer — тот, кто умеет **поставить эксперимент**, который его ответ либо подтвердит, либо опровергнет.

---

## B. Из чего рождается эксперимент

```text
1. Наблюдение     — увидел что-то странное в коде / данных / поведении.
2. Вопрос         — почему так? как сравнить с альтернативой? что если?
3. Гипотеза       — твоё догадка, как ответить.
4. Минимальный    — какой простейший тест отделит "да" от "нет"?
   эксперимент
5. Метрики        — какие числа сделают тест осмысленным.
6. Falsifiability — что бы я увидел, если гипотеза НЕВЕРНА?
   (без этого нет науки, есть confirmation bias)
7. Risk           — что может сломать или скомпрометировать тест?
```

Если у тебя есть наблюдение → ты можешь делать эксперимент. Если нет — сначала погружайся (sprints, code reading), наблюдения появятся.

## C. Как формулировать гипотезу

Плохая гипотеза:
```text
"Bitmap быстрее SQL"
```
- Без класса задач.
- Без числа.
- Без условий, при которых она ложна.

Хорошая гипотеза:
```text
"Для аудиторий, у которых predicate-set состоит ТОЛЬКО из дискретных
включений (no range, no S3-loaded hash), intersect двух заранее
построенных RoaringBitmap-ов даёт p95 latency < 100ms на dataset
до 10M client_id, тогда как текущий путь даёт > 5s.
Гипотеза опровергается, если: (a) латентность bitmap > 500ms, или
(b) build time bitmap > 30 минут на полный rebuild — это сделает
operational cost неприемлемой."
```

Хорошая гипотеза имеет:
- **Класс случая** (для чего применима).
- **Конкретное число** (что считаем выигрышем).
- **Условия фальсификации** (когда говорим «не работает»).

## D. Дизайн минимального эксперимента

```text
1. Что фиксировать (controls):
   - hardware
   - data volume
   - cache state (cold / warm)
   - другие нагрузки кластера

2. Что варьировать (independent variable):
   - один параметр за раз, не несколько

3. Что измерять (dependent variable):
   - latency / throughput / cost / correctness

4. Размер выборки:
   - минимум 3-5 прогонов; для статистики — 20+

5. Анализ:
   - p50, p95, stddev
   - сравнение с baseline через одинаковую методику

6. Воспроизводимость:
   - запиши команды, конфиги, версии
   - чтобы коллега через 3 месяца повторил
```

## F. Категории экспериментов, которые тебе захочется проводить

(Не как «обязательные», а как направления, в которых наблюдения часто появляются. Сам решай, что делать.)

```text
1. Замер baseline текущего поведения.
   Без него любое улучшение — без точки отсчёта.

2. Decomposition end-to-end на компоненты.
   Где время. Где ошибки. Где саtuation.

3. Сравнение двух implementation одной задачи.
   Текущий путь vs альтернатива.

4. Capacity test — что станет при 10x нагрузки.

5. Failure injection — что станет при потере ноды / сбое S3 / медленном CH.

6. Refactor without behavior change.
   Чтобы убедиться, что упрощение не меняет результаты.

7. Counter-factual — что было бы, если бы убрали X.
   Иногда дешевле всего: shadow-mode без X.
```

## H. Антипаттерны экспериментирования

```text
1. Эксперимент без наблюдения = «давайте попробуем».
   → Собери сначала факт, потом ставь эксперимент.

2. Эксперимент с двумя меняющимися переменными.
   → Только одна за раз. Иначе не разберёшь, что на что повлияло.

3. Один прогон.
   → Минимум 3, лучше 5+, для шумных метрик — 20+.

4. Сравнение в разные часы / разную нагрузку.
   → Контроль среды.

5. Игнорирование correctness.
   → «Быстрее» без «правильно» = бесполезно.

6. Перед публикацией изменения.
   → Эксперимент в shadow-mode на проде, прежде чем реально включать.

7. Confirmation bias: видишь только подтверждения.
   → Falsifiability должна быть в гипотезе. Активно ищи опровержение.

8. Не зафиксировать numbers сразу.
   → Через неделю забудешь.
```
